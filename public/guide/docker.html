📘 What is Docker?

Docker is a platform that provides virtual containers on which an application can be deployed independent of the underlying OS of the server. The container can be created from a replica called docker image which contains all the dependencies and can run on any OS that has docker engine, with similar results.

🔄 Virtualization vs Containerization

Virtualization

Virtualization is the process of sharing hardware resources across several virtually isolated and mutually independent systems. It is achieved using a hypervisor which acts as a bridge between the Operating System of each virtual machine and the underlying hardware.

Key Characteristics

  • Applications run on host OS on top of hypervisor
  • Each VM has its own full OS
  • Heavy resource consumption
  • Slower startup times

Containerization

Containerization refers to the process of creating virtually isolated instances of applications along with required dependencies called containers. Containers are independent of the underlying OS and are portable from one OS to another.

Key Advantages

  • Lightweight - shares host OS kernel
  • Fast startup (seconds vs minutes)
  • Portable across different environments
  • Better resource utilization
  • Consistent development to production environment

🧩 Docker Components

Docker Daemon (dockerd) ▼

Docker daemon (dockerd) listens for Docker API requests and manages Docker objects such as images, containers, networks, and volumes. A daemon can also communicate with other daemons to manage Docker services.

Key Functions:

  • Manages container lifecycle
  • Builds images from Dockerfiles
  • Handles networking between containers
  • Manages data volumes
Docker Client ▼

The Docker client (docker) is the primary way that users interact with Docker. When you use commands such as docker run, the client sends these commands to dockerd, which carries them out. The docker command uses the Docker API.

$ docker run nginx $ docker build -t myapp . $ docker ps
Docker Registry ▼

A Docker registry stores Docker images. Docker Hub is a public registry that anyone can use, and Docker is configured to look for images on Docker Hub by default.

Popular Docker Registries:

  • Docker Hub: https://hub.docker.com (Public)
  • AWS ECR: Amazon Elastic Container Registry
  • Google GCR: Google Container Registry
  • Microsoft ACR: Azure Container Registry
  • Harbor: Open-source cloud native registry
Docker Objects (Images, Containers, Networks, Volumes) ▼

Images

A read-only template with instructions for creating a Docker container. Images are built in layers - each instruction in a Dockerfile creates a new layer.

Containers

A runnable instance of an image. Container is the runtime instance of our application with all dependencies.

Networks

Enable communication between containers and the outside world. Provide isolated network environments.

Volumes

Used to persist and share data between containers and the host machine. Data survives container lifecycle.

⚙️ Docker Installation (Linux)

Prerequisites

  • AWS account (or any cloud provider)
  • Linux VM (Amazon Linux AMI recommended)
  • SSH client (MobaXterm, PuTTY, or terminal)

Step-by-Step Installation

Step 1: Create Linux Virtual Machine ▼
  1. Login to AWS account
  2. Navigate to EC2 Dashboard
  3. Click "Launch Instance"
  4. Select Amazon Linux AMI
  5. Choose instance type (t2.micro for testing)
  6. Configure security group (allow SSH port 22)
  7. Launch and download key pair
Step 2: Connect to Linux VM ▼

Use MobaXterm, PuTTY, or terminal to SSH into your instance:

$ ssh -i your-key.pem ec2-user@your-instance-ip
Step 3: Install Docker ▼
# Update system packages $ sudo yum update -y # Install Docker $ sudo yum install docker -y # Start Docker service $ sudo service docker start # OR $ sudo systemctl start docker # Enable Docker to start on boot $ sudo systemctl enable docker # Add ec2-user to docker group $ sudo usermod -aG docker $USER # Exit terminal $ exit # Restart session (Press 'R' in MobaXterm) # Verify Docker installation $ docker info $ docker --version

✅ Installation Complete!

If you see Docker version and system info, installation was successful!

🚀 Essential Docker Commands

View Images

Display all Docker images on your machine

$ docker images

Pull Image

Download an image from Docker Hub

$ docker pull <image-name>

Example:

$ docker pull nginx

Run Container

Create and start a container from an image

$ docker run <image-name>

With port mapping:

$ docker run -p 8080:80 nginx

List Running Containers

Display all active containers

$ docker ps

All containers (including stopped):

$ docker ps -a

Stop Container

Stop a running container gracefully

$ docker stop <container-id>

Remove Container

Delete a stopped container

$ docker rm <container-id>

Force remove:

$ docker rm -f <container-id>

Remove Image

Delete a Docker image

$ docker rmi <image-id>

Force remove:

$ docker rmi -f <image-id>

System Cleanup

Remove all stopped containers, unused images, and networks

$ docker system prune -a

⚠️ Warning: This will remove ALL unused Docker resources!

View Logs

Check container logs

$ docker logs <container-id>

Follow logs:

$ docker logs -f <container-id>

Execute Commands

Run commands inside a container

$ docker exec -it <container-id> /bin/bash

Type 'exit' to leave container

Inspect Container

View detailed container information

$ docker inspect <container-id>

Container Stats

Monitor container resource usage

$ docker stats

💡 Pro Tips

  • Use -d flag to run containers in detached mode (background)
  • Use --name to give containers meaningful names
  • Use -p host:container for port mapping
  • Use -v for volume mounting
  • Press CTRL+C to stop foreground containers

📝 Dockerfile - Building Custom Images

A Dockerfile is a text file containing instructions to build a Docker image. Docker engine processes these instructions from top to bottom using DSL (Domain Specific Language) keywords.

Image Build Process

Dockerfile
⬇️
docker build
⬇️
Docker Image
⬇️
docker run
⬇️
Running Container

Dockerfile Keywords

FROM - Base Image

Specifies the base image to use. This is the starting point of your image.

FROM java:jdk-1.8 FROM python:3.9 FROM node:14 FROM ubuntu:20.04

MAINTAINER / LABEL - Image Metadata

MAINTAINER is deprecated. Use LABEL instead to specify author information.

# Deprecated (old way) MAINTAINER DRONA <contact@dronaintelligence.com> # Recommended (new way) LABEL maintainer="DRONA <contact@dronaintelligence.com>" LABEL version="1.0" LABEL description="My Application"

COPY - Copy Files

Copies files from source to destination during image creation.

COPY <source> <destination> # Examples: COPY target/app.war /app/tomcat/webapps/app.war COPY package.json /app/ COPY . /app

ADD - Advanced Copy

Like COPY, but can also download from URLs and extract archives.

ADD <source> <destination> ADD <URL> <destination> # Examples: ADD app.tar.gz /app/ # Automatically extracts ADD https://example.com/file.zip /app/

Difference: COPY is preferred for local files. ADD has extra features but use with caution.

RUN - Execute Commands (Build Time)

Executes commands during image build. Can have multiple RUN instructions.

RUN yum install maven -y RUN yum install git -y RUN git clone https://github.com/example/repo.git RUN mvn clean package # Best practice: Chain commands to reduce layers RUN yum install maven git -y && \ git clone https://github.com/example/repo.git && \ mvn clean package

CMD - Default Command (Run Time)

Specifies the command to run when container starts. Only last CMD is executed.

CMD ["java", "-jar", "app.jar"] CMD sudo start tomcat # If multiple CMD, only last one runs! CMD echo "First" # Ignored CMD echo "Second" # This one runs

ENTRYPOINT - Container Entry Point

Like CMD, but cannot be overridden. Defines the executable for the container.

ENTRYPOINT ["echo", "Welcome to DRONA Intelligence"] ENTRYPOINT ["java", "-jar", "app.jar"] # Difference: CMD can be overridden at runtime # ENTRYPOINT cannot be overridden

WORKDIR - Set Working Directory

Sets the working directory for subsequent instructions.

WORKDIR /app # All commands after this run from /app RUN pwd # Prints /app COPY . . # Copies to /app

ENV - Environment Variables

Sets environment variables inside the container.

ENV JAVA_HOME=/usr/lib/jvm/java-8 ENV PATH=$PATH:$JAVA_HOME/bin ENV APP_PORT=8080 ENV NODE_ENV=production

ARG - Build Arguments

Defines variables that can be passed at build time.

ARG branch=master ARG version RUN git clone -b $branch https://github.com/example/repo.git # Build with: $ docker build --build-arg branch=develop -t myapp .

EXPOSE - Document Port

Documents which port the container will listen on (metadata only).

EXPOSE 8080 EXPOSE 3000 EXPOSE 80 443 # To actually publish ports, use -p flag: $ docker run -p 8080:8080 myapp

VOLUME - Data Persistence

Specifies mount points for persistent storage.

VOLUME /app/data VOLUME ["/var/log", "/var/db"]

USER - Set User

Sets the user for subsequent instructions.

USER appuser # All commands after this run as appuser

Complete Example Dockerfile

# Sample Dockerfile demonstrating all keywords FROM ubuntu:20.04 LABEL maintainer="DRONA <contact@dronaintelligence.com>" LABEL version="1.0" # Build arguments ARG APP_VERSION=1.0 # Environment variables ENV APP_HOME=/app ENV APP_PORT=8080 # Install dependencies (combine RUN commands) RUN apt-get update && \ apt-get install -y python3 python3-pip && \ apt-get clean # Set working directory WORKDIR $APP_HOME # Copy application files COPY requirements.txt . RUN pip3 install -r requirements.txt COPY . . # Create volume mount point VOLUME /app/data # Expose port EXPOSE $APP_PORT # Set user (security best practice) USER appuser # Container entry point ENTRYPOINT ["python3", "app.py"]

Building & Running Images

# Build image from Dockerfile $ docker build -t myapp . # Build with custom Dockerfile name $ docker build -f Dockerfile.prod -t myapp . # Build with build arguments $ docker build --build-arg APP_VERSION=2.0 -t myapp . # Run the image $ docker run -d -p 8080:8080 --name mycontainer myapp # Tag image $ docker tag myapp username/myapp:v1.0 # Push to Docker Hub $ docker login $ docker push username/myapp:v1.0

🎯 Best Practices

  • Layer Optimization: Combine RUN commands to reduce image size
  • Use .dockerignore: Exclude unnecessary files from build context
  • Multi-stage Builds: Use for smaller production images
  • Security: Run as non-root user when possible
  • Cache: Order instructions from least to most frequently changing

🚀 Docker with Real Applications

Spring Boot Application with Docker ▼

What is Spring Boot?

Spring Boot is an open-source Java framework for building applications quickly. It comes with an embedded server and packages as JAR/WAR files.

Dockerfile for Spring Boot

FROM openjdk:11 COPY target/spring-boot-app.jar /usr/app/ WORKDIR /usr/app/ ENTRYPOINT ["java", "-jar", "spring-boot-app.jar"]

Complete Deployment Steps

# 1. Install Git and Maven $ sudo yum install git maven -y # 2. Clone repository $ git clone https://github.com/dronaintelligence/spring-boot-app.git $ cd spring-boot-app # 3. Build application $ mvn clean package # 4. Build Docker image $ docker build -t spring-boot-app . # 5. Run container (detached mode) $ docker run -d -p 8080:8080 --name myapp spring-boot-app # 6. Check logs $ docker logs myapp # Access: http://your-server-ip:8080

Detached Mode (-d flag)

Running with -d flag runs container in background, allowing you to execute other commands while container runs.

Java Web Application (WAR file) with Docker ▼

Java web applications package as WAR files and need a web server like Apache Tomcat.

Dockerfile for Java Web App

FROM tomcat:8.0.20-jre8 COPY target/maven-web-app.war /usr/local/tomcat/webapps/maven-web-app.war

Deployment Steps

$ git clone https://github.com/dronaintelligence/maven-web-app.git $ cd maven-web-app $ mvn clean package $ docker build -t maven-web-app . $ docker run -d -p 8080:8080 maven-web-app # Access: http://server-ip:8080/maven-web-app/

Note: The WAR file name becomes the context path. Here "maven-web-app.war" creates "/maven-web-app/" URL path.

🌐 Docker Networking

Docker provides flexible networking to facilitate communication between containers and external networks.

Network Types

Network Type Description Use Case
Bridge (Default) Containers on same network communicate via IP addresses or names Default network, single-host communication
Host Container shares host network stack Maximum performance, no port mapping needed
Overlay Multi-host communication in swarm mode Distributed applications across multiple hosts
Macvlan Assigns MAC address, makes container appear as physical device Legacy apps requiring physical network presence
None No network access Isolated containers, maximum security

Network Commands

Create Network

$ docker network create <network-name>

List Networks

$ docker network ls

Inspect Network

$ docker network inspect <network-name>

Connect Container

$ docker network connect <network> <container>

Disconnect Container

$ docker network disconnect <network> <container>

Remove Network

$ docker network rm <network-name>

Remove Unused Networks

$ docker network prune

Create Overlay Network

$ docker network create --driver overlay <name>

Create Bridge Network

$ docker network create --driver bridge <name>

Practical Example

# Create custom network $ docker network create bank-network # List networks $ docker network ls # Inspect network details $ docker network inspect bank-network # Run container on custom network $ docker run -d --network bank-network --name app1 nginx # Run another container on same network $ docker run -d --network bank-network --name app2 nginx # Containers can now communicate using container names! # From app1: ping app2 (will work) # Disconnect container from network $ docker network disconnect bank-network app1 # Remove network $ docker network rm bank-network

Key Benefits

  • Isolation: Containers on different networks can't communicate
  • Name Resolution: Use container names instead of IPs
  • Security: Control which containers can talk to each other
  • Flexibility: Connect/disconnect containers dynamically

💾 Docker Volumes - Data Persistence

Stateless vs Stateful Containers

Stateless Containers Stateful Containers Data lost when container is destroyed Data persists even after container deletion Default Docker behavior Achieved using Docker Volumes Suitable for temporary data Required for databases, file storage

Types of Volumes

1. Anonymous Volumes

Created without a name, Docker generates random ID

$ docker run -v /app/data nginx

2. Named Volumes (Recommended ✅)

Created with a specific name, easy to manage

$ docker volume create my-volume $ docker run -v my-volume:/app/data nginx

3. Bind Mounts

Maps host machine directory to container directory

$ docker run -v /host/path:/container/path nginx

Volume Commands

Create Volume

$ docker volume create <volume-name>

List Volumes

$ docker volume ls

Inspect Volume

$ docker volume inspect <volume-name>

Remove Volume

$ docker volume rm <volume-name>

Remove All Volumes

$ docker system prune --volumes

Remove Dangling Volumes

$ docker volume rm $(docker volume ls -q -f dangling=true)

Practical Example: MySQL with Persistent Storage

Complete MySQL Volume Lab ▼
# 1. Create volume $ docker volume create mysql_data # 2. Run MySQL with volume $ docker run -d --name mysql \ -e MYSQL_ROOT_PASSWORD=root \ -v mysql_data:/var/lib/mysql \ mysql:8.0 # 3. Connect to MySQL $ docker exec -it mysql bash $ mysql -u root -p # 4. Create database and insert data CREATE DATABASE testdb; USE testdb; CREATE TABLE users (id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(50)); INSERT INTO users (name) VALUES ('Alice'), ('Bob'), ('Charlie'); SELECT * FROM users; exit exit # 5. Destroy container $ docker rm -f mysql # 6. Create new container with SAME volume $ docker run -d --name mysql_new \ -e MYSQL_ROOT_PASSWORD=root \ -v mysql_data:/var/lib/mysql \ mysql:8.0 # 7. Verify data still exists! $ docker exec -it mysql_new bash $ mysql -u root -p USE testdb; SELECT * FROM users; # Data is still there! ✅ exit exit # 8. Cleanup $ docker rm -f mysql_new $ docker volume rm mysql_data

🎯 What We Proved

Data persists across container lifecycles when using volumes. This is essential for databases and stateful applications!

🎼 Docker Compose - Multi-Container Applications

Docker Compose is a tool for defining and running multi-container Docker applications. With Compose, you use a YAML file to configure your application's services, networks, and volumes. Then, with a single command, you create and start all the services from your configuration.

Docker Compose Workflow

docker-compose.yml
⬇️
docker-compose up
⬇️
Web Service
Database
Cache

Why Docker Compose?

Key Benefits

  • Single Command Deployment: Start entire application stack with one command
  • Environment Consistency: Same configuration across dev, test, and production
  • Service Isolation: Each service runs in its own container
  • Easy Scaling: Scale individual services independently
  • Network Management: Automatic service discovery and networking
  • Volume Management: Persistent data storage across container restarts

Installation

Docker Compose Installation Methods ▼

Method 1: Using curl (Linux)

# Download Docker Compose $ sudo curl -L "https://github.com/docker/compose/releases/latest/download/docker-compose-$(uname -s)-$(uname -m)" -o /usr/local/bin/docker-compose # Add execution permission $ sudo chmod +x /usr/local/bin/docker-compose # Verify installation $ docker-compose --version

Method 2: Using pip (Python)

# Install using pip $ pip install docker-compose # Verify installation $ docker-compose --version

Method 3: Docker Desktop (Windows/Mac)

Docker Compose comes pre-installed with Docker Desktop for Windows and Mac.

Docker Compose File Structure

Basic YAML Structure
